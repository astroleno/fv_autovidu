# Stage 0 · 歧义告警细则 · Ambiguity Rubric v1.0

**状态：🔒 Phase 1 Prompt 骨架同步（2026-04-17）**
**定位：ScriptNormalizer LLM 端的"何时该报歧义、报到哪个粒度、置信度怎么标"的唯一判据**
**与 01_schema.json 的关系**：本文件对应 schema 中 `ambiguity_report[]` 的 `type` 枚举（6 类），为每一类列出触发条件、证据要求与置信度口径。
**与 00 计划的关系**：展开 §3.6 歧义告警的设计哲学（"歧义优于臆测"，见 §二.4）；不新增类型，不重释类型语义。

---

## 一、核心原则（一句话）

> **宁可报歧义被人工一眼看出"太保守"，也不要编一个看起来合理的答案骗过 EditMap。**

Stage 0 的存在价值就是"事实归一化"，在任何无法由规则或上下文唯一确定的条目上，都必须进 `ambiguity_report`，而不是靠语言模型"编平"。

### 1.1 何时必须报（硬门）

- 候选项 ≥ 2 且没有 ≥ 0.8 置信度区分手段
- 原文使用了松散代词（"她""那女人""那人"）但上下文存在 ≥ 2 个可指对象
- 原文时间表达模糊（"过了一会儿""良久""那一年"）且未能与已知场次锚点对齐
- 原文动作主语省略且前后句主语不同
- 场景切换无显式锚点（时间/地点/人物全未变，但叙事口吻转折）

### 1.2 何时不报（避免过度告警）

- 单候选（上下文只有 1 个可指对象）→ 直接写 `character_registry.pronoun_bindings`，不进 `ambiguity_report`
- 原文显式标注（"苏无名说"）→ 直接使用，不报
- 已在 `bible.characters[].aliases` 中登记的别名 → 直接合并，不报

## 二、六类歧义触发清单

### 2.1 `pronoun_resolution` · 代词消解

**触发条件**（任一命中）：
- 代词出现时上下文 ≤ 3 句内有 ≥ 2 个**同性别**候选角色
- 代词与上一主语间有场景切换或新角色引入
- 代词与最近角色提及间隔 > 5 段落

**证据要求**：
- `seg_id`：代词所在段 ID（必填）
- `candidates`：列全部可指 CHAR_ID，顺序按距离近→远
- `picked`：当前采信的 CHAR_ID
- `rationale`（可选，走 `detail`）：说明采信理由（距离/语义/性别）

**置信度口径**：

| 区间 | 判据 | 是否建议人工复核 |
|-----|-----|---------------|
| `≥ 0.85` | 最近句主语明确 + 性别单一命中 | 否 |
| `0.6 ~ 0.85` | 最近候选距离 ≤ 3 句 + 性别命中 | 否 |
| `0.4 ~ 0.6` | 多候选等距 + 性别无法区分 | **是** |
| `< 0.4` | 完全靠语义猜测 | **是** |

### 2.2 `alias_merge` · 别名合并

**触发条件**（任一命中）：
- 剧本出现某称谓（"夫人""总裁""神探"）但 `bible.characters[].aliases` 未登记
- 同一角色在同一场内被 ≥ 3 个不同别名引用
- 别名本身是一个尚未入 Bible 的新角色候选

**证据要求**：
- `seg_id`：别名首次出现段
- `candidates`：候选 CHAR_ID 列表（允许只有 1 个候选，此时表示"疑似新角色"）
- `picked`：要么已有 CHAR_ID，要么新建占位 CHAR_UNKNOWN_<hash>
- `detail` 中必须写清"是并入已有角色还是建议新建"

**置信度口径**：

| 区间 | 判据 | 是否建议人工复核 |
|-----|-----|---------------|
| `≥ 0.9` | 别名已在 Bible aliases 列表 | 否 |
| `0.6 ~ 0.9` | 语义强相关（同场只出现一次某称谓） | 否 |
| `< 0.6` | 新建占位角色 | **是** |

**回写约定**：置信度 ≥ 0.9 的别名发现，Stage 0 **应**把它加入 `character_registry[CHAR_X].aliases_in_script`，同时进 `ambiguity_report` 供 Bible 维护者回写 Bible（见 00 §九 风险表 "Bible alias 表不全"）。

### 2.3 `time_mode_uncertain` · 时间模式不确定

**触发条件**（任一命中）：
- 原文出现"那一年""多年以前""她还小""记得那天"等回忆触发词，但无明确日期锚点
- 镜头进入某人的内心视角且下一场不回到当前时间线
- 平行叙事（两线交替）但两线均以"现在时"语气写

**证据要求**：
- `beat_id`：时间模式不确定的 beat
- `candidates`：`time_mode` 候选（如 `["present", "flashback"]`）
- `picked`：当前采信的 time_mode
- `detail`：引用原文触发短语

**置信度口径**：

| 区间 | 判据 | 是否建议人工复核 |
|-----|-----|---------------|
| `≥ 0.8` | 存在明确时间锚点（"五年前""去年冬天"） | 否 |
| `0.6 ~ 0.8` | 存在回忆触发词 + 下一场回到现在 | 否 |
| `< 0.6` | 仅靠语气推断 | **是** |

**默认值规则**（D3）：置信度 < 0.6 时 `picked = "present"`，并把真实候选放进 `candidates`。

### 2.4 `scene_continuity` · 场景连续性

**触发条件**（任一命中）：
- 原文无显式场次标号，但时间/地点/人物其中两项发生变化
- 同一空间内两段对白被一个"——"或"···"分隔，无法判断是否跨时
- 场景切换同时伴随 time_mode 变化（如"现在→回忆"）

**证据要求**：
- `seg_id`：切换点所在段（两段的分界）
- `candidates`：`["continuous", "new_scene"]`
- `picked`：采信的判断
- `detail`：说明三要素（时间/地点/人物）哪些变了

**置信度口径**：

| 区间 | 判据 | 是否建议人工复核 |
|-----|-----|---------------|
| `≥ 0.85` | 三要素中 ≥ 2 项明显变化 | 否 |
| `0.6 ~ 0.85` | 一项变化 + 叙事口吻转折 | 否 |
| `< 0.6` | 纯靠语气推断 | **是** |

**影响**：`picked=new_scene` → `scene_timeline` 新建一条；`picked=continuous` → 合并入当前场。

### 2.5 `prop_holder_unknown` · 道具持有者不明

**触发条件**（任一命中）：
- 道具从某 beat 起脱离上一持有者但未明确新持有者（如"他把怀表放下"之后无人再提）
- 多人场景中出现"桌上的刀"但未说明是谁的
- 道具在场景外被提及（"他说起那枚戒指"）

**证据要求**：
- `beat_id`：道具状态变化所在 beat
- `seg_id`：线索所在段（与 `evidence_seg` 对齐）
- `candidates`：候选 holder（可以含 `null` 表示"场景持有"）
- `picked`：采信者
- `detail`：描述脱手过程

**置信度口径**：

| 区间 | 判据 | 是否建议人工复核 |
|-----|-----|---------------|
| `≥ 0.8` | 下一 beat 内出现明确新持有者 | 否 |
| `0.5 ~ 0.8` | 场景延续且道具未再提及，默认 `In_Scene` + `holder=null` | 否 |
| `< 0.5` | 道具跨场消失或异常出现 | **是** |

### 2.6 `action_attribution` · 动作归属

**触发条件**（任一命中）：
- 动作动词主语省略且前一句主语与动作语义不匹配（"她低头，转身走了"中"转身走了"的主语是谁？）
- 互动动作双方均有可能发起（"两人对视"后"一拳挥来"）
- 动作句的反应主体与动作发起者角色冲突

**证据要求**：
- `seg_id`：动作所在段
- `candidates`：候选 actor 列表
- `picked`：采信的 actor
- `detail`：说明推断路径

**置信度口径**：

| 区间 | 判据 | 是否建议人工复核 |
|-----|-----|---------------|
| `≥ 0.85` | 前句主语即为本句合理主语 | 否 |
| `0.6 ~ 0.85` | 语义推断主语（如"他挥拳"对应前句攻击方） | 否 |
| `< 0.6` | 纯猜 | **是** |

## 三、置信度口径总表（跨类型统一规范）

| 置信度带 | 含义 | `suggest_human_review` 默认值 | `beat_ledger[].ambiguity_flags` 是否引用 |
|---------|------|-----------------------------|---------------------------------------|
| `≥ 0.85` | 高置信，Stage 0 可直接写真值字段 | false | 是（可选，用于审计） |
| `0.6 ~ 0.85` | 中置信，写入真值字段但同时登记歧义 | false | **必须** |
| `0.4 ~ 0.6` | 低置信，真值字段按 `picked` 写入但置位 review | **true** | **必须** |
| `< 0.4` | 极低置信，`picked` 只能是候选之一，严禁编造 | **true** | **必须** |

**交叉约束**：
- `ambiguity_report[].suggest_human_review=true` → EditMap 消费时视为"未消解"，回退原文推理（与 00 §六 ⑤ 追加 prompt 对齐）
- `beat_ledger[].ambiguity_flags` 引用的每一条 ID 必须存在于 `ambiguity_report[]`（schema 不做引用校验，引擎层 assert）
- 单 beat 的 `ambiguity_flags.length > 3` → 记 `_warnings.beat_high_ambiguity`，建议 pipeline 层在 mode=heavy 时启动人工复核门

## 四、ID 命名与引用

- `AMB_<NNN>`：三位零填充数字，按在当前包内的 `ambiguity_report` 数组顺序递增（`AMB_001, AMB_002 ...`）
- 跨包不保证唯一；合并多剧本场景由 pipeline 层重编号
- `AMB_*` 不得复用删除的 ID，删除视为"永久墓碑"（审计需要）

## 五、反臆测底线（LLM 端硬规则）

在 ScriptNormalizer 的 Prompt 里会显式复述，但以本文件为**唯一判据**：

1. **不得**在 `candidates` 为空的情况下写任何 `ambiguity_report` 条目（没有候选就没有歧义，本身是一个 bug）
2. **不得**对同一 seg/beat 的同类型歧义生成 > 1 条记录（必须合并为一条）
3. **不得**为了降低 `ambiguity_report.length` 而把 `confidence` 人为提到 0.85 以上（如被回归对齐出现"假高置信"模式，`mode=heavy` 时强制每 beat 至少一次歧义复核 · 见 00 §七）
4. **不得**在 `ambiguity_report` 中写入未列入上述 6 类的 `type` 值（schema 枚举已约束，LLM 若生成非法值由 ajv 拒绝，pipeline 走 §九 失败兜底）
5. **不得**把导演化判断（镜头/光影/情绪基调）作为歧义类型报告（违反 00 §二 职责边界）
