## 一、为什么 editmap/ 切片不走 injection_map（架构决策）

### 1.1 EditMap 的角色：pipeline 的「路由器」

EditMap 在 SD2Workflow v5 中的独特地位：

| 阶段 | 消费的路由标签来源 | 产出的路由标签 |
|------|-------------------|---------------|
| **EditMap** | 无（本身就是路由器） | `block_index[i].routing.{structural, satisfaction, psychology, shot_hint, paywall_level}` |
| **Director** | 读 EditMap 产出 → `injection_map.director.conditional.*` | Director 自身产出（markdown + `continuity_out`） |
| **Prompter** | 读 `meta.video.aspect_ratio` → `injection_map.prompter.conditional.*` | 四段切 SD2 prompt |

**EditMap 是路由器，Director / Prompter 是路由器的消费者**——两者所需的方法论触达方式也必须不同。

### 1.2 死循环问题：为什么 editmap/ 不能走 injection_map

如果 `editmap/` 切片通过 `injection_map.yaml` 条件加载，会形成：

```
识别 satisfaction 母题  ←  加载 subtext_and_signals.md
       ↑                         ↓
       └─────  需要识别后才能加载的死循环  ──────┘
```

即使改为 `mode: always` 也有问题——`always` 语义是「每次 consumer 调用必注入」，但 EditMap 的「方法论总包」不是 7 份零碎切片的叠加，而是一份**完整的编剧认知框架**，应当作为 system prompt 的常量组成部分，而非动态拼装物。

### 1.3 决策：静态拼接（static_concat）

pipeline 在构建 EditMap system prompt 时，由 `scripts/sd2_pipeline/call_editmap_sd2_v5.mjs` 直接执行：

```js
// 伪代码（真实实现在 fv_autovidu/scripts/sd2_pipeline/call_editmap_sd2_v5.mjs）
const editmapSlicesDir = path.join(slicesRoot, 'editmap');
const slices = fs.readdirSync(editmapSlicesDir)
  .filter(f => f.endsWith('.md'))
  .sort();  // 固定顺序见 §三
const methodology = slices
  .map(f => fs.readFileSync(path.join(editmapSlicesDir, f), 'utf8'))
  .join('\n\n---\n\n');
const systemPrompt = [editMapMainPrompt, methodologyHeader, methodology].join('\n\n');
```

**特性**：
- 所有 6 份切片**全量可见**，无截断风险；
- 不占 `injection_map.yaml` 的 `max_total_tokens_per_consumer` 预算；
- 切片间**无路由依赖**，彼此引用（如 `hook_strategies.md` 引用 `dramatic_action.md §4.3`）可被 LLM 自由跨引；
- `routing_trace` **不记录** editmap/ 切片（它们不是「注入产物」，是 prompt 的静态组成部分）。

## 三、拼接顺序与预算约束

### 3.1 拼接顺序（固定）

为保证 EditMap LLM 调用可重现，editmap/ 切片按以下**固定顺序**拼入 system prompt：

```
[EditMap 主 prompt（含 §0~§9 的铁律 + 输出 schema 骨架）]
    ↓
【编剧方法论（静态挂载 · 本节下列 6 份按顺序拼接）】
    1. dramatic_action.md        —— 先给「戏剧动作」最小单位（所有后续判断的基石）
    2. character_want_need.md    —— 再给「人物」维度（决定主角 ID 与矛盾特征）
    3. subtext_and_signals.md    —— 再给「视听化 + 爽点信号」（decide routing.satisfaction）
    4. hook_strategies.md        —— 再给「首 block 专用」钩子框架（决定 routing.structural[0]）
    5. two_track_pacing.md       —— 再给「全片节奏」双轨（决定 emotion_loops）
    6. proof_and_info_gap.md     —— 最后给「结构性账本」（最高抽象层，覆盖全片）
    ↓
[schema 合同（07 冻结契约精简版） + 一次性自检清单]
```

**顺序决策依据**：从「最小单位 → 人物 → 视听化 → 首 block → 全片节奏 → 结构性账本」，**尺度由小到大递进**，让 LLM 在读到结构性账本时已具备前 5 份切片提供的语义基础。

### 3.2 token 硬限 ≤ 12,000

**为什么设硬限**：防止「方法论原文粗放搬入」导致 EditMap system prompt 失控（过长会挤压 `scriptContent` 的 context window）。

**硬限数字由来（v5.0 实测定板）**：
- 6 份切片 v5.0 GA 瘦身后实测合计 **~10,981 tokens**，最高单份 `proof_and_info_gap.md` **~2,353 tokens**；
- 再压会牺牲方法论骨架完整性（三问 / 映射表 / 反例任一删减都会让 LLM 失去判定依据）；
- 硬限 12,000 = 实测 10,981 + **~1,000 tokens buffer**，覆盖未来轻度补充（如 `retracted` 悬疑剧新增示例、新版爽点信号条目）；
- 该数字与 EditMap 主 prompt（~15,350 tokens）+ user message（~1,800 tokens）合计 ≤ 30,000 tokens，对 Claude 200k / GPT-4o 128k / Gemini 1M 均在 15% 以内。

**如何保障**：
- 单份切片 ≤ 2,400 tokens（估算）；
- 6 份合计 ≤ 12,000 tokens（含分隔符与挂载头部元信息）；
- CI 在 `tests/regression/` 阶段用 `wc -m` × 0.6 做近似量度，超出 → PR 不合入（见 `06_v5-验收清单与回归基线.md §八`）。

**超限应对**：优先拆切片、不优先压正文。若单份切片超 2,400 tokens，应按主题拆成两份微切片（如 `proof_and_info_gap.md` 可拆 `info_gap.md` + `proof_ladder.md`）而非精简表述。

## 五、与 Stage 0 · ScriptNormalizer 的正交关系

### 5.1 Stage 0 的职责：事实归一化，不是方法论路由

根据 `docs/stage0-normalizer/00_ScriptNormalizer-v1-计划.md §二` 的设计原则：

> Stage 0 只回答「剧本里到底发生了什么 / 谁参与 / 持续多久」，**不回答「怎么拍」、不产出 routing 标签**。

因此 Stage 0 与 editmap/ 切片的关系是**严格正交**的：

| 维度 | Stage 0（ScriptNormalizer） | editmap/ 切片 |
|------|---------------------------|---------------|
| 做什么 | 事实归一化（指代统一、beat 切分、双时间轴、状态账本） | 导演化判断前的**编剧认知框架**（戏剧动作 / 弧光 / 爽点信号 / 钩子 / 节奏 / 证据链） |
| 何时触发 | EditMap 之前（Stage 0 → EditMap） | EditMap system prompt 构建时（静态拼接） |
| LLM 负担 | 给 EditMap 减负：指代已消解、beat 已切好、时长已估算 | 给 EditMap 配备**叙事层的认知工具**（LLM 读完可直接做 routing 判断） |
| 是否影响路由 | 否（Phase 1 红线，不产出 routing 标签） | 否（本身是判断 routing 的依据，但自己不被路由） |

### 5.2 Phase 1（v5.1 候选）：两者完全独立

Stage 0 Phase 1 的 5 件事（`character_registry / scene_timeline / beat_ledger / temporal_model / state_ledger`）**都不触及 editmap/ 切片**：

- Stage 0 输入：`scriptContent + assetManifest + episodeDuration`
- Stage 0 输出：`normalizedScriptPackage`
- EditMap 输入：`scriptContent + assetManifest + episodeDuration + (可选) normalizedScriptPackage`
- EditMap system prompt：`editMapMainPrompt + editmap/*.md（静态拼接）`

**结论**：Stage 0 上线后，editmap/ 切片挂载机制**完全不变**。两者在输入端合流（都是 EditMap 的入口），但知识挂载路径正交——Stage 0 走「输入数据」，editmap/ 切片走「system prompt 常量」。

### 5.3 Phase 2（v5.2+）远景：editmap/ 切片仍维持静态挂载

即便 Stage 0 升格（将 `character_registry / state_ledger` 纳入 `meta.*` 硬字段）后，editmap/ 切片**仍不走 injection_map**。理由：

1. editmap/ 切片的内容是**叙事层的元认知**（「什么是戏剧动作」「什么是 Show Don't Tell」），而不是「已发生事实」的描述，天然属于 system prompt 的组成部分；
2. 若将来 EditMap 主 prompt 需要瘦身（把重复的铁律表格移入独立切片），瘦身后的切片也应走静态拼接，不走 routing——同理，EditMap 是路由器。

**唯一例外**：若未来 v6+ 引入多个 EditMap 变体（例如「悬疑专用 EditMap」「情感向 EditMap」），可在 system prompt 构建阶段**按变体静态选择不同切片集合**——这仍然是静态选择，不是 routing。

## 六、脱敏要求（与 `01_v5-知识库清洗与脱敏规范.md` 对齐）

### 6.1 切片正文

editmap/ 切片的正文**不得出现**下列任何一类内容（与通用切片一致，外加 editmap 专属加强项）：

- 任何外部作品 / 编剧书 / 导演名 / 经典段落引用的具体指称（如「《XX》开场」「XX 大师说」等）；
- 任何能反推来源的路径（`reference/山音*.md`、`docs/短剧/*` 等）；
- 具体电影片名的中英文原名；
- 作家 / 编剧 / 导演的真实姓名。

### 6.2 `.gitignore` 配合

外部编剧方法论原文（如「山音」）已在 `.gitignore` 中屏蔽：

```
reference/山音*.md
reference/山音*
```

见 `01_v5-知识库清洗与脱敏规范.md §七` 追加项 W0-7。

### 6.3 追溯表只写职责

`docs/v5/_traceability.yaml` 中 editmap/ 条目只登记 **slice 承担的职责**（如 `concept: 开场钩子五策略`），**不写** `traced_from`（因为 editmap/ 是 v5.0 本地化方法论，虽然借鉴了外部资料但已完成深度重写，不需要追溯字段）。示例：

```yaml
- v5_artifact: 4_KnowledgeSlices/editmap/hook_strategies.md
  concept: 开场钩子五策略 + 15s 生死测试
  responsibilities:
    - 驱动 block_index[0].routing.structural 的 hook_block / cold_open 打标
    - 驱动 psychology_plan[0].group == "hook"
    - 驱动 diagnosis.notes 的 hook_missing / hook_15s_test_failed 登记
  # traced_from 字段刻意留空——editmap/ 切片均为 v5.0 本地化深度改写
```
