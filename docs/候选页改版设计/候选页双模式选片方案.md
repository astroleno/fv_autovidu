# 候选页双模式选片方案

## 1. 设计目标

候选页同时承载两类任务：

1. 总览：尽可能高密度地浏览所有镜头，快速判断进度、状态和哪些镜头需要处理。
2. 选片：聚焦单个镜头，在多个候选之间快速比较并做出选择。

这两类任务对界面的要求是冲突的，因此不再强行用同一套布局兼顾，而是拆成双模式。

## 2. 模式定义

页面保留同一路由与同一份数据，但提供两种模式：

1. `overview`：列表模式
2. `picking`：选片模式

两个模式共享：

1. `SceneGroup` 分组
2. 筛选条件
3. 候选选定逻辑
4. 重新生成 / 自定义参数 / 精出能力

差异只体现在布局、信息密度和交互方式。

### 2.1 模式切换入口

模式切换必须是显式、低学习成本的：

1. 列表模式顶部提供 `Overview / Picking` 切换按钮。
2. 列表模式下每个镜头卡片提供“进入选片”入口。
3. 列表模式下支持 `Enter` 进入当前聚焦镜头的选片模式。
4. 选片模式顶部提供“返回列表”入口。
5. 选片模式支持 `Esc` 返回列表模式。

切换规则：

1. 从列表模式进入选片模式时，默认进入当前点击镜头。
2. 若用户通过顶部“进入选片模式”切换，则默认进入当前筛选结果中的第一个“待选镜头”。
3. 返回列表模式时，保留原有筛选条件和滚动位置。

## 3. 列表模式 Overview

### 3.1 定位

列表模式只负责“扫全局”，优先在一屏里展示尽可能多的镜头。

### 3.2 展示内容

每个镜头卡片保留高密度信息：

1. 镜头号
2. 状态
3. 运镜 / 时长 / 画幅
4. 主候选或首帧缩略
5. 已选定状态
6. 候选数量

### 3.3 不展开的内容

列表模式不完整展示：

1. 首尾帧详细对比
2. 长提示词文本
3. 完整资产区

### 3.4 作用

列表模式解决：

1. 哪些镜头还没选
2. 哪些镜头无视频
3. 哪些镜头需要进入选片模式详细处理

## 4. 选片模式 Picking

### 4.1 定位

选片模式专门服务于连续快速决策，一次聚焦当前镜头。

### 4.2 页面结构

选片模式不是“放大版列表卡片”，而是聚焦型工作台。布局权重必须明确：

1. 顶部：模式切换、当前位置、当前筛选上下文
2. 中间：候选视频主体区
3. 侧边或下方：首尾帧、提示词、资产、配音状态
4. 底部：键盘提示、上一镜头 / 下一镜头

核心原则：

1. 当前镜头必须足够突出
2. 候选视频必须大于列表模式
3. 参考信息只展示当前镜头，不展示其它镜头噪音

### 4.3 布局优先级

桌面端必须满足以下权重：

1. 候选区占主要视觉空间，建议占据主体宽度的 65%~75%。
2. 参考区作为辅区，建议占据主体宽度的 25%~35%。
3. 当前激活候选的视频高度不应低于 `480px`，或不低于可视区高度的约 50%。
4. 参考区允许滚动，但候选区优先保持稳定可见。

实现上可以采用：

1. 左大右小布局：左侧候选区，右侧参考区。
2. 或上大下小布局：上方候选区，下方参考区。

但不允许：

1. 候选区与参考区视觉权重接近。
2. 候选视频缩小成接近列表模式尺寸。
3. 选片模式退化成“另一种密排卡片列表”。

## 5. 键盘交互规则

### 5.1 候选选择

1. `1`：选中第 1 个候选
2. `2`：选中第 2 个候选
3. `3`：选中第 3 个候选
4. `4`：选中第 4 个候选

规则：

1. 按下数字键后，立即将对应候选设为当前镜头的已选候选。
2. 被选中的候选立即成为当前激活候选。
3. 当前激活候选自动循环播放。

补充规则：

1. 数字键优先对应当前页面可见的前 4 个候选。
2. 若当前镜头候选数超过 4 个，数字键仍只直达前 4 个候选。

### 5.2 镜头切换

1. `ArrowLeft`：切换到上一镜头
2. `ArrowRight`：切换到下一镜头

规则：

1. 切换镜头前，系统先提交当前激活候选为当前镜头的最终选择。
2. 提交成功后再切换镜头。
3. 提交失败则停留当前镜头并提示错误。

### 5.2.1 撤销机制

选片是高频操作，必须允许快速纠错。

首版定义：

1. `Ctrl+Z` / `Cmd+Z`：撤销最近一次“候选选择”操作。
2. 撤销只回退最近一次选中结果，不回退镜头切换。
3. 若最近一次操作是“离开镜头时自动提交默认候选”，也允许撤销。
4. 撤销成功后，恢复上一个已选候选，并将其设为当前激活候选。

约束：

1. 撤销栈仅需支持当前会话内最近若干次操作，不要求跨刷新持久化。
2. 首版不做复杂历史面板。

### 5.3 默认选择规则

用户需求是：

1. 如果不切换其它候选，直接进入下一组，则当前默认候选就是最终选择。

因此定义如下：

1. 若镜头已有已选候选，则它就是默认激活候选。
2. 若镜头尚未选定，但存在候选，则第 1 个候选为默认激活候选。
3. 进入镜头时，默认激活候选自动播放。
4. 离开镜头时，若用户未手动改选，则默认激活候选照样提交为最终选择。

### 5.4 播放规则

1. 当前激活候选自动播放
2. 播放结束后自动循环
3. 切换到其它候选时，新激活候选立即开始播放并循环
4. 非激活候选不自动播放

### 5.5 边界情况

1. 镜头只有 2 个候选时，按 `3` / `4` 无效果
2. 镜头无候选时，数字键无效果
3. 第一镜头按 `ArrowLeft` 无效果
4. 最后一镜头按 `ArrowRight` 无效果，或提示已到结尾

### 5.6 候选数量大于 4 的规则

必须明确 5+ 候选的交互，避免规范不完整。

首版定义：

1. 页面最多同时突出展示前 4 个候选。
2. 若后端返回 5+ 候选，其余候选进入“更多候选”区域。
3. `Tab` / `Shift+Tab` 用于在全部候选间循环切换当前激活候选。
4. 当前激活候选无论来自前 4 个还是“更多候选”，都可通过 `Enter` 再次确认选中。
5. 数字键 `1/2/3/4` 不扩展到第 5 个及以后候选。

设计取舍：

1. 保留数字键的直接性，不把规则做复杂。
2. 5+ 候选不是主流路径，因此接受它的效率略低于前 4 个候选。

## 6. 选片模式展示内容

### 6.1 候选区

候选区为视觉主体：

1. 1 个候选：大尺寸单列
2. 2 个候选：双列
3. 3 个及以上候选：最多双列，不强压成三列

每个候选显示：

1. 视频
2. 候选序号
3. 模型 / 分辨率 / mode / seed
4. 是否已选中
5. 是否为当前激活候选

候选区视觉要求：

1. 当前激活候选必须有明显高亮。
2. 已选中与“当前激活”是两个不同状态，视觉上要能区分。
3. 若已选中候选就是当前激活候选，则合并为最强高亮态。

### 6.2 参考区

参考区展示当前镜头的最小参考集：

1. 首帧 / 尾帧
2. `visualDescription`
3. `videoPrompt`
4. `imagePrompt`
5. 资产标签
6. 配音状态摘要

文本优先级：

1. `visualDescription`
2. `videoPrompt`
3. `imagePrompt`

超长文本默认截断，可展开。

### 6.3 进度定义

顶部进度文案必须避免歧义，统一使用两组信息：

1. 当前位置：`当前第 X / Y 个镜头`
2. 完成进度：`已选 Z / Y`

不使用单独的 `12 / 48` 裸数字，避免用户分不清是“当前序号”还是“已完成数量”。

## 7. 数据与状态规则

### 7.1 已选候选

每个镜头只能有一个已选候选。

写入时机：

1. 按数字键时立即写入
2. 离开镜头时写入当前激活候选

### 7.1.1 自动提交的例外

如果当前镜头没有任何候选，则切换镜头时不做自动提交。

如果当前镜头存在候选但写入失败，则：

1. 停留当前镜头
2. 保留当前激活候选状态
3. 弹出明确错误提示

### 7.2 无候选镜头

无候选镜头下：

1. 不存在默认激活候选
2. 数字键无效
3. 左右键仍可切换镜头
4. 保留“重新生成视频 / 自定义参数”入口

### 7.3 是否支持批量选择

本期不支持批量选择或“连续镜头套用同一候选规则”。

原因：

1. 当前设计目标是先把单镜头快速决策路径做扎实。
2. 批量选择会显著增加状态复杂度和误操作风险。
3. 该能力可作为后续迭代单独评估，不应混入首版核心流程。

## 8. 组件建议

建议新增或拆分：

1. `VideoPickModeToggle`
2. `VideoPickFocusPanel`
3. `VideoPickCandidateGrid`
4. `VideoPickReferencePanel`
5. `useVideoPickKeyboard`

说明：

1. 以上是职责建议，不要求机械拆成 5 个新文件。
2. 若现有组件可自然承载其中某部分职责，可优先扩展而非盲目新增组件。
3. `overview` 继续沿用现有轻量卡片体系，`picking` 使用新的聚焦型组件。

## 9. 实现优先级

### P0

1. 页面双模式切换
2. 选片模式单镜头聚焦布局
3. `1/2/3/4` 选择候选
4. `ArrowLeft/ArrowRight` 切镜头
5. 离开镜头时提交当前激活候选
6. 激活候选自动循环播放

### P1

1. 自动定位到第一个待选镜头
2. 顶部显示明确进度：`当前第 X / Y 个镜头` + `已选 Z / Y`
3. 长文本展开
4. 模式记忆
5. 仅浏览待选镜头
6. `Ctrl+Z` / `Cmd+Z` 撤销最近一次选择

### P2

1. 精出快捷键
2. 镜头详情快捷入口
3. “更多候选”区的更高效浏览方式
4. 批量选择能力评估

## 10. 可访问性与范围说明

### 10.1 可访问性

虽然这是键盘优先工作流，但仍需满足基础可访问性：

1. 当前激活候选必须有明确焦点态。
2. 模式切换按钮、候选卡、上一镜头 / 下一镜头都必须可被键盘聚焦。
3. 屏幕阅读器应能读出当前镜头号、候选序号、是否已选中、是否当前激活。
4. 高亮不只依赖颜色，需同时有边框、标签或图标差异。

### 10.2 移动端范围

本方案首版默认面向桌面端键盘工作流。

1. 平板和移动端不是首版重点目标。
2. 若需要在移动端访问，优先保证可浏览与可点击，不要求完整复刻键盘效率。
3. 不因为移动端适配需求削弱桌面端的选片效率。

### 10.3 模式记忆粒度

模式记忆按“项目或剧集级”生效，不按单镜头记忆。

1. 用户上次在该剧集使用 `overview`，下次默认仍进入 `overview`。
2. 用户上次在该剧集使用 `picking`，下次默认仍进入 `picking`。
3. 但无论默认进入哪个模式，都应恢复到上次的筛选条件。

## 11. 预期收益

双模式后，候选页职责更清晰：

1. 列表模式负责看全局
2. 选片模式负责快决策
3. 大量镜头时，键盘流能显著提高选片效率
4. 撤销与明确边界规则降低误操作焦虑
